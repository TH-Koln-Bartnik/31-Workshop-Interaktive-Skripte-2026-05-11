## Lessons Learned — für die nächste Skript-Erstellung

### 1. Quarto-YAML hat brutale Edge Cases

**Erste Zeile MUSS exakt `---` sein.** Tresorit/OneDrive-Sync oder Auto-Format-Tools setzen manchmal versehentlich `# ---` als ersten Zeilenanfang. Das macht den ganzen YAML-Header zu einem Markdown-Kommentar. Quarto rendert das Dokument dann lautlos als plain HTML statt als Reveal — die Folie sieht aus wie eine Webseite, nicht wie ein Deck. **Erste Diagnose** bei „warum sind meine Slides keine Slides": die erste Zeile prüfen.

**Deutsche Anführungszeichen `„…"` innerhalb einer doppelt-gequoteten YAML-Zeile brechen den Parser**, weil das schließende deutsche `"` (U+201C) als Ende des YAML-Strings interpretiert wird. Symptom: der Wert wird nach dem ersten deutschen `"` abgeschnitten, der Rest gilt als kaputtes YAML. Lösung: äußere Anführungszeichen als Single Quotes setzen, dann ist der Inhalt egal:

```yaml
footer: 'Workshop „Mein Titel" · TH Köln'
```

**`prefer-html: true` löst den Plotly+PDF-Konflikt nicht zuverlässig.** In zwei Quarto-Versionen (1.5 und 1.6) bricht der PDF-Render trotzdem mit „Functions that produce HTML output found in document targeting pdf output". Pragmatischer Sofort-Fix: PDF aus `_quarto.yml` raus, HTML-only. Saubere Langzeit-Lösung: interaktive Inhalte (Plotly, Observable, Leaflet) in eigene HTML-Dateien in `interactions/` auslagern und per `<iframe>` einbinden — dann ist die `.qmd` frei von HTML-Widgets und PDF läuft. Diese Stolperfalle ist im Skill `quarto-publishing` und im Tutor-Prompt dokumentiert.

### 2. Reveal.js-Styling braucht eigene Selektoren

Eine `styles.scss`, die für das HTML-Buch sauber funktioniert (Selektoren wie `body`, `h1`, `a`), hat **null Wirkung auf Reveal-Slides**. Reveal wickelt jede Folie in einen `<div class="reveal">`-Container, dessen eigene Default-Styles höhere CSS-Spezifität haben. Lösung: alle Brand-Selektoren mit `.reveal`-Präfix nochmal eintragen, im selben File:

```scss
.reveal h1 { color: var(--th-red); }
.reveal a, .reveal a:visited { color: var(--th-red); }
.reveal ul li::marker { color: var(--th-magenta); }
.reveal code { color: var(--th-red); background: var(--gray-100); }
.reveal pre { border-left: 3px solid var(--th-magenta); }
```

Der finale `styles.scss`-Stand im Workshop hat diesen Block am Ende — beide Output-Formate sind damit abgedeckt.

Reveal kennt zwei SCSS-Layer-Marker: `/*-- scss:defaults --*/` für SASS-Variablen (muss VOR den Rules stehen) und `/*-- scss:rules --*/` für CSS-Regeln. Wer das verdreht, bekommt „expected `}`"-Fehler beim SCSS-Compile.

`theme: [default, ../styles.scss]` in der Reveal-YAML bedeutet: Default-Theme als Basis, eigene SCSS überlagert. Reihenfolge ist wichtig — die eigene Datei muss zuletzt geladen werden.

### 3. Tresorit/OneDrive/Dropbox-Sync produziert systematisch Probleme

**Truncation mid-write.** Datei wird halbgespeichert: Header da, Inhalt fehlt. Symptome reichen von „Div unclosed"-Fehler im Quarto-Render bis zu fehlenden Klammern in SCSS („expected `}`"). Lösung: Datei im Editor öffnen, einmal Strg+S drücken, eine Sekunde warten, neu rendern. Wenn das mehrfach passiert: Projektordner aus dem Cloud-Sync herausziehen in `C:\Users\...\Documents\GitHub\`.

**Race Condition bei Resource-Copy.** Quartos `resources: ["interactions/*"]` wird beim Render evaluiert. Wenn eine Datei zu dem Zeitpunkt sync-bedingt nicht erreichbar ist, wird sie übersprungen — ohne Warnung. Symptom: nach dem Push fehlt eine HTML-Datei in `docs/interactions/`, der iFrame im Browser zeigt 404. Genau das ist heute mit `lernspiel-kapitel1.html` passiert (gefixt durch manuelles Kopieren in den docs-Ordner). Sofort-Fix: Datei manuell aus `interactions/` nach `docs/interactions/` kopieren und committen. Vorbeugung: vor dem Render kurz `ls interactions/*.html` prüfen, dass alle erwarteten Dateien sichtbar sind.

**Bash-Sandbox vs. Read-Tool sehen unterschiedliche Versionen.** Im Cowork-Setup hat jede Tool-Klasse eine eigene Sicht aufs Filesystem. Read und Edit wirken auf die Live-Windows-Datei, Bash sieht eine cache-verzögerte Linux-Mount-Sicht. Das hat heute mehrfach zu falsch-positiven Fehlerdiagnosen geführt. Faustregel: bei Diskrepanzen Read-Tool als Wahrheit nehmen, Bash für ausführbare Befehle.

### 4. Quotation Drift — die unterschätzte Fehlerklasse

Wenn Sie ein Label im Skript ändern (z. B. Überschrift „End-Test" → „Reaktivierung: Aktiv auslaufen lassen"), entgeht jeder Grep-basierten Suche das Vorkommen, das dieses Label im Lauftext **zitiert** (etwa in der Index-Sektion „Wie Sie dieses Skript benutzen"). Die Such-Ersetzen-Mechanik findet nur Treffer der buchstäblichen Stelle, nicht jede Erwähnung. Diese Klasse ist nur durch reines Lesen zu finden.

Der Reviewer-Persona-Prompt im Skript greift genau dieses Problem auf: „Linse 2 — Quotier-Drift" prüft jedes Mal, wenn der Lauftext ein anderes Element zitiert (eine Überschrift, ein Knopflabel, eine Datei), ob das Zitierte noch genau in dieser Form existiert. Ein paar Lese-Minuten finden mehr als zwei Stunden Greppen.

### 5. Visual QA für Slides — der PPT-Skill-Workflow funktioniert auch für Reveal

Ablauf: rendern → einzelne Folien als PNG screenshotten → mit frischen Augen inspizieren → Liste der Befunde → fixen → erneut screenshotten. In der Cowork-Sandbox machbar, indem man Quarto (Tarball-Download, ca. 130 MB) und Chromium über Puppeteer (`node node_modules/puppeteer/install.mjs`) installiert. Anschließend ein kleines Puppeteer-Skript, das per `Reveal.slide(idx, 0, -1)` durch die Folien navigiert, alle Fragmente vorklickt (`Reveal.nextFragment()`) und pro Folie ein PNG erzeugt. Die Bilder unter `outputs/screenshots/` ablegen, dann via Read-Tool als Bild öffnen und prüfen.

Das findet Layout-Probleme zuverlässig: zu große Schrift, URL-Wraps in vier Zeilen, überlappende Elemente, fehlende Bilder, kontrastarme Texte auf farbigen Hintergründen. Ohne diese visuelle Schleife sind das Befunde, die der Workshop-Teilnehmende live findet — peinlicher als nötig.

### 6. Per-Kapitel-Interaktionsvielfalt schlägt monotone Wiederholung

Wenn jedes Kapitel den selben Drag-and-Drop-Quiz-Bauplan hat, stumpfen die Reaktivierungen ab. Besser: Form folgt Inhalt. Cross-Refs lernt man am besten durch Hot-Spot-Klick auf echtes Code-Snippet, Stil-Wirkungen durch Vorher-Nachher-Slider, Citation-Syntax durch Cloze, Vorhersagen durch Slider-Prediction (das psychologisch wirksamste Format nach Fiorella & Mayer 2016), Misskonzepte durch Wahr/Falsch mit sofortiger Begründung, Reihenfolge-Wissen durch Sortier-Aufgabe. Der Bonus: jede Reaktivierung ist gleichzeitig ein Showcase eines Interaktionstyps, den die Teilnehmenden in ihrem eigenen Skript nachbauen können — Meta-Lehre durch das Format selbst.

### 7. Galerien als „Fülle der Möglichkeiten" — die Verstehen-Entscheiden-Umsetzen-Gestalten-Prüfen-Architektur

Wenn ein Kapitel viele Beispiele zeigen will, lohnt sich eine **explizite didaktische Progression** statt einer alphabetischen oder zufälligen Liste. In Kap. 7 sind die zehn Galerie-Lernspiele in fünf Gruppen sortiert: erst „Verstehen" (Grundprinzipien aktivieren), dann „Entscheiden" (Optionen abwägen), dann „Umsetzen" (technische Grundlagen üben), dann „Gestalten" (selbst etwas designen), zuletzt „Prüfen" (Qualität sichern). Diese Logik ist gleichzeitig die Architektur, die die Teilnehmenden in ihrem eigenen interaktiven Skript abbilden können — Meta-Lehre noch eine Stufe höher als reine Form-Vielfalt. Bei jeder Galerie pro Eintrag: Ein-Satz-Einführung in Italic, dann der iFrame, dann zur nächsten Gruppe weitergehen.

### 8. Phasen-Architektur senkt kognitive Last spürbar

Eine Tagesablauf-Tabelle in drei Zeilen (eine pro Phase, Inhalt grob skizziert, kein Stündlich-Plan) macht den Workshop weniger einschüchternd als eine Acht-Block-Tabelle mit Minutengenauigkeit. Wichtig: in jeder Phase **höchstens ein neues Werkzeug**. Phase 1 nutzt nur RStudio plus den KI-Tutor-Tab. Phase 2 fügt Zotero hinzu — RStudio bleibt vertraut, das Neue ist klein und isoliert. Phase 3 fügt GitHub plus Pages dazu, und der Veröffentlichungs-Moment am Tagesende belohnt den ganzen Tag.

Der Quickstart spiegelt das auch im Installations-Set: Phase-1-Tools sind Pflicht für 09:00 (Quarto, R, RStudio, ChatGPT-Tutor), Phase-2-Tools können in der Mittagspause kommen (Zotero, Better BibTeX), Phase-3-Tools erst zum Nachmittag (GitHub-Konto, GitHub Desktop). Das senkt die Vor-Installations-Hürde dramatisch.

Zeitangaben für „X Minuten Klickarbeit" sind aus dem Quickstart raus — sie demotivieren und stimmen je nach Hardware ohnehin nicht.

### 9. Etymologie als Eselsbrücke wirkt didaktisch

Begriffe wie „Quarto" (lat. *quartus*, das alte Buchformat), „YAML" (rekursives selbstironisches Akronym), „RStudio" (nach R, das nach Robert Gentleman und Ross Ihaka heißt), „HTML" (Ted Nelsons Hypertext-Idee 1965, Berners-Lee am CERN), „Zotero" (albanisch *zotëroj* = beherrschen), „Git" (britischer Slang, Linus Torvalds Selbstironie) sitzen besser, wenn ihre Geschichte mitgeliefert wird. Das Multimedia-Prinzip in eigener Sache — Wort plus zweite Repräsentation, hier in Form einer Mini-Anekdote.

### 10. „Reaktivierung: Aktiv auslaufen lassen" statt „End-Test"

Die Umbenennung der Schluss-Übung pro Kapitel hat zwei Effekte: erstens nimmt sie den Klausurdruck raus (es ist kein „Test"), zweitens framet sie die Übung explizit als aktives Abrufen — Retrieval Practice ohne das Wort zu nennen. Didaktisch geframte Begriffe wirken nachweislich anders als neutrale.

### 11. Strang-Konzept (A/B/C) entlastet die Inhaltserfindung

Damit die Teilnehmenden nicht parallel zu allen technischen Schritten auch noch Beispiel-Inhalte erfinden müssen, gibt das Skript drei thematische Stränge zur Wahl: A (Wissenschaftliche Einstellung), B (Recherche und Quellenqualität), C (Transparent schreiben), alle aus der Bartnik-Palmer-Vorlage „Projektorientierte Einführung in kritisches wissenschaftliches Denken" (in Anlehnung an Bergstrom & West, *Calling Bullshit*). Jedes Mach-mit-Beispiel hat strangbezogene Sub-Vorlagen. Das spart Mental Load und macht die Beispiele übergreifend kohärent.
